# Policies no Domínio

## 1. O que é uma policy

`Policy` é um artefato de domínio usado para representar uma regra de negócio reutilizável, normalmente quando:

- a regra não pertence claramente a uma única entidade;
- a regra precisa ser aplicada em mais de um fluxo;
- a regra combina múltiplas condições de negócio.

## 2. Para que é usado

Policies são usadas para:

1. centralizar regras de negócio transversais;
2. evitar duplicação de regra entre entidades e casos de uso;
3. explicitar critérios de permissão/bloqueio no domínio;
4. manter entidades e use cases mais coesos.

## 3. Padrões comuns de implementação

### 3.1 `check()`

- retorna `true/false`;
- útil quando o fluxo decide o que fazer em cada resultado.

### 3.2 `assert()`

- lança erro de domínio quando a regra não é satisfeita;
- útil para bloquear imediatamente operações inválidas.

Observação: nem toda policy precisa ter os dois métodos, mas esse é um padrão comum e prático.

## 4. Quando usar policy (e quando não usar)

### 4.1 Use policy quando

- a regra é de negócio e reaproveitável;
- a regra cruza mais de um conceito de domínio;
- a regra deve ser lida como “pode/não pode”.

### 4.2 Não use policy quando

- a validação pertence apenas ao estado interno de uma entidade;
- a lógica é puramente técnica (infra, transporte, ORM, API externa);
- a regra só existe em um único ponto e não ganha clareza extra como policy.

## 5. Exemplos corretos

### 5.1 Exemplo com `assert` (alinhado ao projeto)

```ts
export class CanReemitirNotaPolicy {
    static check(venda: VendaFiscalDTO): {
        canReemit: boolean;
        reason: string | null;
    } {
        const resValidateVenda = this.validateVenda(venda);
        if (!resValidateVenda.canEmit) {
            return {
                canReemit: false,
                reason: resValidateVenda.reason,
            };
        }
        return {
            canReemit: true,
            reason: null,
        };
    }

    static assert(venda: VendaFiscalDTO, resumoFiscal: ResumoFiscal | null) {
        const results = this.check(venda);

        if (!results.canReemit) throw new AppValidationError(results.reason!);
    }

    private static validateVenda(venda: VendaFiscalDTO): {
        canEmit: boolean;
        reason: string | null;
    } {
        if (!venda.dataFinalizacao) {
            return {
                canEmit: false,
                reason: "Apenas vendas finalizadas podem ter documentos fiscais reemitidos",
            };
        }

        if (venda.dataCancelamento) {
            return {
                canEmit: false,
                reason: "Venda cancelada não pode ser reemitida",
            };
        }
        return {
            canEmit: true,
            reason: null,
        };
    }
}

```

### 5.2 Exemplo com `check` + `assert`

```ts
export class CanApplyDiscountPolicy {
  static check(input: { desconto: number; valorItem: number }): boolean {
    return input.desconto >= 0 && input.desconto <= input.valorItem;
  }

  static assert(input: { desconto: number; valorItem: number }): void {
    if (!this.check(input)) {
      throw new ValidationDomainError("Desconto inválido para o valor do item");
    }
  }
}
```

## 6. Exemplo incorreto

```ts
export class ProdutoAtivoPolicy {
  static assert(produto: Produto): void {
    if (!produto.ativo) {
      throw new ValidationDomainError("Produto inativo não pode ser vendido");
    }
  }
}
```

Problemas:

- a regra depende apenas do estado interno de `Produto`;
- essa decisão pertence à própria entidade (`Produto`) e não a uma policy separada;
- espalhar regras internas da entidade em policies reduz coesão do modelo;
- aumenta a chance de duplicação de regra em vários pontos.

## 7. Checklist rápido para policies

1. a policy representa regra de negócio e não detalhe técnico?
2. a regra é reutilizável e faz sentido estar centralizada?
3. a policy está livre de ORM, HTTP e dependências de infraestrutura?
4. o tipo de retorno (`check` ou `assert`) está adequado ao uso?
5. os erros lançados são erros de domínio?
