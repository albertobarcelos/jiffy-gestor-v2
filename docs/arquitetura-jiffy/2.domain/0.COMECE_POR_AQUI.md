# Camada de Domínio - Comece por Aqui

Este documento é uma visão geral da camada de **domínio** no Jiffy.  
Aqui estão as regras que definem o negócio: o que é válido, o que é inválido, o que pode e o que não pode acontecer no sistema.

## 1. O que é a camada de domínio

A camada de domínio é o núcleo da aplicação dentro da arquitetura limpa.

Ela concentra:

- regras de negócio invariáveis;
- conceitos centrais do sistema;
- políticas que independem de framework, banco, API externa ou detalhes técnicos.

Em termos práticos, o domínio representa a linguagem do negócio, não a linguagem da infraestrutura.

## 2. Pelo que o domínio é responsável

### 2.1 Regras e invariantes de negócio

- definir o que torna uma entidade válida;
- garantir consistência de estado;
- impedir transições inválidas.

Exemplos:

- o que é um produto válido;
- quais campos são obrigatórios para cadastrar/atualizar um produto;
- quais são os status possíveis de uma venda.

### 2.2 Modelagem de conceitos

- formalizar conceitos do negócio em entidades e objetos de valor;
- encapsular validações e comportamentos que pertencem ao conceito.

Exemplos:

- o que é um cliente válido;
- qual a regra para fechamento de caixa;
- quando uma operação é permitida ou bloqueada por política de negócio.

## 3. Pelo que o domínio não é responsável

O domínio **não** deve orquestrar fluxos de aplicação e integrações técnicas.

Exemplo de orquestração que não pertence ao domínio:

1. verificar se usuário existe;
2. verificar se já existe convite pendente;
3. persistir dados no banco;
4. enviar e-mail.

Esse tipo de coordenação é responsabilidade da camada de **aplicação**.

## 4. Regras obrigatórias para implementação no domínio

### 4.1 Pureza da camada

- o domínio deve ser puro;
- não pode depender de camadas externas;
- não pode conhecer detalhes de infraestrutura.

### 4.2 Dependências

- evitar dependências externas desnecessárias;
- bibliotecas utilitárias só devem ser usadas quando não violarem o isolamento do domínio.

Exceção controlada: validação com bibliotecas como `zod`, quando fizer sentido no contexto e sem acoplamento a framework.

## 5. Como navegar nesta pasta

- `1.ENTITIES.md`: entidades e identidade de negócio.
- `2.VALUE_OBJECTS.md`: objetos de valor e imutabilidade.
- `3.DOMAIN_SERVICES.md`: serviços de domínio.
- `4.PORTS.md`: contratos/portas para interação com o exterior.
- `5.VALIDATION.md`: estratégia de validação no domínio.
- `6.POLICIES.md`: políticas e regras transversais de negócio.
