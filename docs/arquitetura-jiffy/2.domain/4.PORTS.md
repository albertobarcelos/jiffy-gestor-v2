# Ports (Contratos) na Arquitetura

## 1. O que é uma port

`Port` é um contrato (interface, tipo ou classe abstrata) que define uma capacidade esperada por uma camada, sem acoplar essa camada à implementação concreta.

Em resumo:

- a `port` declara **o que precisa ser feito**;
- o `adapter`/implementação define **como será feito**.

## 2. Onde uma port pode existir

Uma port pode existir em mais de uma camada, dependendo da direção da dependência.

### 2.1 Domain ports

Usadas quando o domínio precisa de algo externo sem conhecer tecnologia.

Exemplos:

- `IAuthenticationRepository`
- `ICardapioRepository`
- `IPagamentoRepository`
- `IOperacaoPdvRepository`
- `Discount` e `Increase` (abstrações de comportamento)

### 2.2 Application ports

Usadas para orquestração de casos de uso, integração entre módulos e gateways de saída da aplicação.

Exemplos de contratos possíveis:

- `INotificationGateway` (enviar e-mail/SMS/push);
- `IFileStorage` (upload/download de arquivos);
- `IEventBus` (publicação de eventos de aplicação).

### 2.3 Infrastructure ports

Também é possível existir contratos internos de infraestrutura para desacoplar tecnologias entre adapters (ex.: provider de fila, cache, cliente HTTP).

Exemplos de contratos possíveis:

- `IQueueProvider`;
- `ICacheProvider`;
- `IHttpClient`.

## 3. Para que é usado

Ports são usadas para:

1. aplicar inversão de dependência;
2. reduzir acoplamento entre regra e tecnologia;
3. permitir troca de implementação com menor impacto;
4. melhorar testabilidade com mocks/fakes/stubs;
5. explicitar fronteiras entre camadas.

## 4. Regras gerais para criar ports

### 4.1 Intenção clara

- nomear contrato e métodos pela capacidade que ele oferece;
- evitar nomes técnicos quando o contrato for de negócio.

### 4.2 Assinaturas estáveis

- usar tipos que façam sentido para a camada dona da port;
- evitar vazar detalhes de implementação na assinatura.

### 4.3 Sem implementação

- port define contrato, não fluxo técnico;
- SQL, Prisma, HTTP, SDK e detalhes de provider ficam no adapter.

## 5. Exemplos por camada

### 5.1 Exemplo de port no domínio

```ts
export interface IProdutoRepository {
  getById(id: string): Promise<Produto | null>;
  save(produto: Produto): Promise<void>;
}
```

### 5.2 Exemplo de port na aplicação

```ts
export interface INotificationGateway {
  sendInviteEmail(input: {
    to: string;
    companyName: string;
    inviteLink: string;
  }): Promise<void>;
}
```

### 5.3 Exemplo de port na infraestrutura

```ts
export interface IHttpClient {
  post<TResponse>(url: string, body: unknown): Promise<TResponse>;
}
```

## 6. Exemplo incorreto

```ts
import { prisma } from "@prismafolder/client.ts";
import axios from "axios";

export class FecharVendaUseCase {
  async execute(input: { vendaId: string; email: string }) {
    const venda = await prisma.venda.findUnique({
      where: { id: input.vendaId },
    });

    if (!venda) throw new Error("Venda não encontrada");

    await prisma.venda.update({
      where: { id: input.vendaId },
      data: { status: "FECHADA" },
    });

    await axios.post("https://email-service/send", {
      to: input.email,
      subject: "Venda fechada",
    });
  }
}
```

Problemas:

- o caso de uso acopla direto com Prisma e HTTP client;
- não existe `port` para persistência nem para notificação;
- troca de tecnologia exige alterar regra de aplicação;
- testes viram integração obrigatória (mais lentos e frágeis);
- fronteiras entre aplicação e infraestrutura ficam quebradas.

## 7. Versão correta do mesmo caso

```ts
// application/ports/IVendaRepository.ts
export interface IVendaRepository {
  getById(id: string): Promise<Venda | null>;
  updateStatus(id: string, status: "FECHADA" | "CANCELADA"): Promise<void>;
}

// application/ports/INotificationGateway.ts
export interface INotificationGateway {
  sendVendaFechadaEmail(input: { to: string; vendaId: string }): Promise<void>;
}

// application/use-cases/FecharVendaUseCase.ts
export class FecharVendaUseCase {
  constructor(
    private vendaRepository: IVendaRepository,
    private notificationGateway: INotificationGateway
  ) {}

  async execute(input: { vendaId: string; email: string }) {
    const venda = await this.vendaRepository.getById(input.vendaId);
    if (!venda) throw new Error("Venda não encontrada");

    await this.vendaRepository.updateStatus(input.vendaId, "FECHADA");
    await this.notificationGateway.sendVendaFechadaEmail({
      to: input.email,
      vendaId: input.vendaId,
    });
  }
}
```

Pontos positivos:

- regra de aplicação depende apenas de contratos;
- Prisma/HTTP ficam nos adapters de infraestrutura;
- troca de tecnologia não exige alterar o caso de uso;
- testes unitários ficam simples com mocks de `ports`.

## 8. Checklist rápido para ports

1. o contrato está na camada certa para a dependência desejada?
2. a assinatura expressa capacidade e não tecnologia?
3. a implementação concreta está fora do contrato?
4. o contrato permite trocar adapter sem alterar regra?
5. o contrato está simples de mockar em testes?
