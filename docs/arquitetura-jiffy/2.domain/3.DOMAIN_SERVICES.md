# Domain Services no Domínio

## 1. O que é um domain service

`Domain Service` é um serviço da camada de domínio usado para encapsular **regra de negócio** que:

- envolve múltiplas entidades;
- não pertence claramente a uma única entidade;
- precisa consultar dados externos ao domínio por meio de contratos (`ports/repositories`), sem levar infraestrutura para dentro da regra.

## 2. Para que é usado

Use domain service para:

1. aplicar regras de negócio transversais entre entidades;
2. centralizar validações de negócio reutilizáveis;
3. manter entidades focadas no próprio ciclo de vida, sem inchar com responsabilidades que são de colaboração.

## 3. Quando usar e quando não usar

### 3.1 Use domain service quando

- a regra é claramente de negócio;
- a regra precisa de colaboração entre agregados/entidades;
- a regra será reutilizada por vários casos de uso;
- a entidade ficaria artificialmente inchada ao receber essa lógica.

### 3.2 Não use domain service quando

- o problema é apenas orquestração de fluxo (isso é da aplicação);
- a lógica é técnica (infraestrutura, transporte, persistência, cache, integração);
- a regra pertence claramente a uma entidade específica.

## 4. Estrutura recomendada no Jiffy

### 4.1 Dependências

- depender de contratos do domínio (ex.: `I...Repository`);
- não depender de implementação concreta de banco/framework;
- receber dependências via injeção no construtor.

### 4.2 Localização

- manter em `domain/services`;
- quando fizer sentido, separar políticas específicas em `domain/services/policies`.

## 5. Exemplo correto

Exemplo comum no projeto: serviço de domínio validando regras e consultando repositório por contrato.

```ts
export class ProdutoDomainService {
  constructor(private cardapioRepository: ICardapioRepository) {}

  async ensureProdutoExists(produtoId: string) {
    const produto = await this.cardapioRepository.getProdutoByIdentifier(produtoId);
    if (!produto) throw new NotFoundDomainError("Produto");
    return produto;
  }

  async ensureCodigoProdutoUniqueness(codigo: string, empresaId: string) {
    const existing = await this.cardapioRepository.getProdutoByCodigo(codigo, empresaId);
    if (existing) throw new ValidationDomainError("Código de produto já existe");
  }
}
```

Pontos positivos:

- regra de negócio centralizada;
- uso de contrato de domínio;
- sem dependência de infraestrutura concreta;
- fácil de reutilizar em múltiplos casos de uso.

## 6. Exemplo incorreto

```ts
export class ProdutoDomainService {
  async criarProdutoEEnviarEmail(input: any) {
    // grava no banco
    // chama API externa
    // envia e-mail
    // publica evento em fila
  }
}
```

Problemas:

- mistura domínio com orquestração de aplicação;
- mistura domínio com infraestrutura;
- baixa coesão e alto acoplamento;
- difícil de testar e reutilizar.

## 7. Checklist rápido para domain services

1. a regra é de negócio e não apenas fluxo técnico?
2. a lógica não cabe naturalmente em uma única entidade?
3. o serviço depende apenas de contratos do domínio?
4. erros lançados são erros de domínio?
5. o serviço está sem detalhes de framework/infra?
