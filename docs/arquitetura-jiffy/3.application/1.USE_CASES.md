# Use Cases na Camada de Aplicação

## 1. O que é um use case

`Use Case` é a unidade de orquestração da camada de aplicação.

Ele representa uma ação do sistema com objetivo claro, por exemplo:

- criar produto;
- aceitar convite;
- autenticar usuário;
- atualizar perfil.

## 2. Para que é usado

Use cases são usados para:

1. coordenar chamadas entre domínio e infraestrutura;
2. organizar o fluxo de execução de uma funcionalidade;
3. aplicar regras de aplicação (ordem, transação, integração);
4. retornar dados prontos para a camada de entrada (normalmente DTOs).

## 3. Estrutura recomendada no Jiffy

### 3.1 Classe + `execute`

- cada use case deve expor um método principal `execute`;
- nome da classe deve refletir a intenção (`Create...UseCase`, `Update...UseCase`, `Login...UseCase`).

### 3.2 Dependências via construtor

- injetar contratos/serviços no construtor;
- evitar instanciar dependências dentro do `execute`;
- utilizar contratos (`ports/repositories`) em vez de implementação concreta.

### 3.3 Fluxo claro

Fluxo típico:

1. validar pré-condições de aplicação;
2. consultar dados necessários;
3. acionar domínio (entidades, VOs, policies, services);
4. persistir/integrações externas;
5. mapear retorno para DTO quando necessário.

### 3.4 Retorno previsível

- retornar tipo explícito;
- evitar `any`;
- quando necessário, mapear entidade para DTO na aplicação.

## 4. O que não colocar em use case

- regra central de domínio que pertence à entidade/value object/policy;
- acesso direto a tecnologia (ORM/SDK/HTTP);
- métodos gigantes sem separação de responsabilidades;
- efeitos colaterais ocultos e difíceis de testar.

## 5. Exemplos corretos

```ts
export class CreateProdutoUseCase {
    constructor(
        private readonly cardapioRepository: ICardapioRepository,
        private readonly cardapioCreationService: CardapioCreationService
    ) {}

    async execute(
        input: CreateProdutoInput,
        empresaId: string
    ): Promise<ProdutoDTO> {
        const { produto, impressorasIds, gruposComplementosIds } =
            await this.cardapioCreationService.buildProduto(input, empresaId);

        const created = await this.cardapioRepository.createProduto(
            produto,
            gruposComplementosIds,
            impressorasIds
        );

        return ProdutoMapper.toDTO(created);
    }
}
```

Outro Exemplo

```ts
export class ConfirmEmailUseCase {
    constructor(
        private readonly authenticationRepository: IAuthenticationRepository,
        private readonly logger: ILoggerService
    ) {}

    async execute(userId: string): Promise<void> {
        // Decode without verifying first to extract the userId
        const usuario =
            await this.authenticationRepository.getUsuarioById(userId);

        if (!usuario) {
            throw new ConstraintError("Usuário não encontrado.");
        }

        if (usuario.emailConfirmado) {
            // Idempotency: if already active, do nothing
            return;
        }

        // Activate user
        usuario.activate();
        usuario.confirmEmail();

        // Save the user
        await this.authenticationRepository.updateUsuario(usuario);

        this.logger.info(
            `E-mail confirmado com sucesso para o userId: ${usuario.id}`,
            {
                requestContext: { userId: usuario.id },
            }
        );

        return;
    }
}

```

Pontos positivos:

- responsabilidade clara;
- dependências injetadas;
- domínio acionado no lugar certo;
- retorno tipado e previsível.

## 6. Exemplo incorreto

```ts
export class CreateProdutoUseCase {
    async execute(input: any) {
        if (!input.nome) throw new Error("Nome obrigatório");

        const produto = {
            ...input,
            valor: input.valor < 0 ? 0 : input.valor,
            criadoEm: new Date(),
        };

        const created = await prisma.produto.create({ data: produto });
        await axios.post("https://api-externa/log", created);

        return created;
    }
}
```

Problemas:

- validação e regra de domínio espalhadas no use case;
- acoplamento direto com Prisma e HTTP;
- ausência de contratos e baixa testabilidade;
- retorno de infraestrutura vazando para fora da aplicação.

## 7. Checklist rápido para use cases

1. o nome do use case descreve uma ação clara do sistema?
2. o método `execute` tem fluxo simples e legível?
3. dependências estão injetadas por contrato?
4. regras centrais ficaram no domínio (e não no use case)?
5. retorno está tipado e consistente?
