# Camada de Aplicação - Comece por Aqui

Este documento é uma visão geral da camada de **aplicação** no Jiffy.  
Aqui fica a coordenação entre entrada, domínio e infraestrutura para entregar funcionalidades do sistema.

## 1. O que é a camada de aplicação

A camada de aplicação é a ponte entre:

- interfaces de entrada (HTTP, jobs, filas, eventos);
- regras de negócio do domínio;
- adapters externos (banco, APIs, mensageria, arquivos e afins).

## 2. Pelo que a aplicação é responsável

### 2.1 Coordenação entre camadas

- receber dados das camadas de entrada;
- acionar regras de domínio de forma organizada;
- encaminhar operações para adapters externos via contratos.

### 2.2 Gestão de fronteiras arquiteturais

- proteger o domínio de detalhes técnicos;
- depender de abstrações/contratos em vez de implementações concretas;
- manter separação clara entre regra, orquestração e infraestrutura.


## 3. Pelo que a aplicação não é responsável

- não definir regras centrais e invariantes do domínio;
- não implementar detalhes técnicos de persistência, transporte ou SDKs;
- não acoplar a aplicação diretamente a tecnologia específica quando houver contrato.

## 4. Regras obrigatórias para implementação

### 4.1 Dependências corretas

- aplicação pode depender de `domain`;
- aplicação depende de **contratos**, não de implementações concretas de infraestrutura;
- domínio não depende da aplicação.

### 4.2 Organização interna

- separar responsabilidades por módulo e contexto;
- manter pontos de entrada e saída explícitos;
- evitar classes ou funções “faz-tudo”.

### 4.3 Tratamento de erros

- preservar erros de domínio como erros de negócio;
- encapsular erros técnicos sem contaminar a linguagem do domínio;
- manter comportamento previsível para camadas consumidoras.

## 5. Checklist rápido para camada de aplicação

1. a camada está protegendo o domínio de detalhes técnicos?
2. as dependências passam por contratos/abstrações?
3. responsabilidades estão separadas entre aplicação, domínio e infraestrutura?
4. há acoplamento indevido com framework, ORM ou SDK?
5. o comportamento está previsível e testável?
