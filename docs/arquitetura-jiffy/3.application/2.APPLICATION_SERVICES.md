# Application Services

## 1. O que é um application service

`Application Service` é um componente da camada de aplicação usado para encapsular lógica de apoio à orquestração.

**IMPORTANTE**: Ele não substitui `use case`, e sim reduz duplicação quando a mesma sequência de passos é usada por múltiplos casos de uso.

## 2. Para que é usado

Application services são usados para:

1. extrair trechos reutilizáveis de fluxo de aplicação;
2. centralizar preparação de dados antes/depois do domínio;
3. evitar duplicação entre vários use cases;
4. manter use cases menores e mais legíveis.

## 3. Diferença entre use case e application service

### 3.1 Use case

- representa uma ação completa do sistema;
- é ponto principal de entrada da funcionalidade;
- possui intenção de negócio explícita (`CreateConvite`, `LoginPdv`, etc.).

### 3.2 Application service

- representa uma capacidade auxiliar reutilizável;
- não é, por si só, a funcionalidade final exposta ao cliente;
- é chamado por um ou mais use cases.

## 4. Quando criar um application service

Crie quando:

1. dois ou mais use cases repetem o mesmo bloco de lógica;
2. a lógica não pertence ao domínio, mas também não é detalhe de infraestrutura;
3. a extração melhora coesão e reduz tamanho dos use cases;
4. o serviço pode ser testado isoladamente com valor claro.

## 5. Regras de implementação no Jiffy

### 5.1 Dependências

- utilizar dependências por contratos;
- não utilizar dependência direta em framework/ORM quando existir adapter/port;
- injetar dependências via construtor.

### 5.2 Escopo de responsabilidade

- manter foco em uma capacidade de aplicação;
- evitar classe genérica “utilitária” sem contexto;
- não mover invariantes de domínio para application service.


## 6. Exemplo correto

Exemplo de serviço de aplicação preparando criação de produto para reutilização:

```ts
export class CardapioCreationService {
  constructor(
    private readonly cardapioRepository: ICardapioRepository,
    private readonly preferenciasRepository: IPreferenciasRepository
  ) {}

  async buildProduto(input: CreateProdutoInput, empresaId: string) {
    // valida relações necessárias
    // resolve ids auxiliares
    // instancia entidade Produto (domínio)
    // retorna dados prontos para o use case persistir
  }
}
```

Uso no use case:

```ts
export class CreateProdutoUseCase {
  constructor(
    private readonly cardapioRepository: ICardapioRepository,
    private readonly cardapioCreationService: CardapioCreationService
  ) {}

  async execute(input: CreateProdutoInput, empresaId: string) {
    const { produto, impressorasIds, gruposComplementosIds } =
      await this.cardapioCreationService.buildProduto(input, empresaId);

    return this.cardapioRepository.createProduto(
      produto,
      gruposComplementosIds,
      impressorasIds
    );
  }
}
```

## 7. Exemplo incorreto

```ts
export class ConviteAppService {
  async run(input: any) {
    // validação de entidade
    // criação de entidade
    // persistência
    // envio de e-mail
    // autorização
    // logging
    // retorno HTTP
  }
}
```

Problemas:

- serviço faz papel de use case, domínio e infraestrutura ao mesmo tempo;
- responsabilidade ampla e pouco coesa;
- alto acoplamento e baixa testabilidade;
- fronteiras arquiteturais ficam confusas.

## 8. Checklist rápido para application services

1. este serviço é realmente reutilizável por mais de um use case?
2. a lógica extraída pertence à aplicação (e não ao domínio)?
3. o use case ficou menor e mais legível após a extração?
4. as dependências continuam desacopladas por contrato?
5. o serviço evita virar um “orquestrador genérico” sem foco?
