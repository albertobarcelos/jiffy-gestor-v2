# Controllers na Camada de Apresentação

## 1. O que é um controller

`Controller` é o componente da apresentação que recebe a requisição e chama a aplicação.

Ele é responsável por adaptar dados de entrada/saída entre protocolo externo e casos de uso internos.

## 2. Para que é usado

Controllers são usados para:

1. ler dados de `params`, `query`, `body` e contexto da requisição;
2. fazer a primeira validação de entrada (formato e tipos primitivos);
3. chamar use cases ou serviços de aplicação com entrada já validada;
4. mapear resultado para resposta HTTP (status, payload, headers);
5. delegar tratamento de erro para o mecanismo padrão da API.

## 3. Como controllers são usados no Jiffy

No Jiffy, é comum o padrão:

1. `make...Controller(req)` para montar dependências;
2. métodos por ação (`create`, `findMany`, `getById`, `update`, `delete`, etc.);
3. uso de `req, res, next` no fluxo Express.

## 4. O que deve ficar no controller

- adaptação da entrada para o formato esperado pela aplicação;
- validação de borda (campos obrigatórios básicos, formato e tipos primitivos);
- uso de validadores da aplicação (ex.: `application/validators`);
- chamada de use case/serviço correto;
- definição de status code e estrutura da resposta.

## 5. O que não deve ficar no controller

- regra de domínio;
- validação de regra de negócio;
- orquestração complexa de múltiplos passos de negócio;
- acesso direto a banco, ORM ou cliente externo;
- lógica duplicada que deveria estar na aplicação.

## 6. Exemplo correto

```ts
import { createClienteInputValidator } from "@/modules/gestor/pessoa/application/validators/validators.ts";

export class ClienteController {
  constructor(private readonly createClienteUseCase: CreateClienteUseCase) {}

  async create(req: Request, res: Response, next: NextFunction) {
    try {
      const validatedInput = createClienteInputValidator.parse(req.body);
      const dto = await this.createClienteUseCase.execute(validatedInput);
      return res.status(201).json(dto);
    } catch (err) {
      return next(err);
    }
  }
}
```

## 7. Exemplo incorreto

```ts
export class ClienteController {
  async create(req: Request, res: Response) {
    const existente = await prisma.cliente.findUnique({
      where: { email: req.body.email },
    });

    if (existente) throw new Error("Cliente já existe");

    const created = await prisma.cliente.create({ data: req.body });
    return res.json(created);
  }
}
```

Problemas:

- controller acoplado ao banco;
- regra de negócio no lugar errado;
- ausência de camada de aplicação como ponto de orquestração;
- baixa testabilidade e manutenção pior.

## 8. Checklist rápido para controllers

1. controller só adapta entrada/saída e chama aplicação?
2. a validação de formato/tipos primitivos está sendo feita no controller?
3. esse validador veio da camada de aplicação (`application/validators`)?
4. há dependência direta com tecnologia de persistência?
5. o tratamento de erro está padronizado com `next(err)`?
6. status code e payload estão coerentes com a ação?
