# Domain Validation

## 1. O que é validação de domínio

Validação de domínio é a garantia de que os dados e estados respeitam as regras do negócio.

Não é apenas validar formato (`string`, `number`), mas garantir invariantes, por exemplo:

- preço não negativo;
- status permitido;
- transição de estado válida;
- entidade sempre em estado consistente.

## 2. Onde a validação pode acontecer no domínio

### 2.1 Entidades

Principal ponto de validação de criação e atualização (`create`, `update`).

### 2.2 Value Objects

Validação do próprio conceito (ex.: `Money`, `Email`, `Cpf`, `Cnpj`).

### 2.3 Domain Services e Policies

Regras que envolvem múltiplas entidades, verificações transversais ou restrições de negócio.

## 3. Padrão adotado no Jiffy

### 3.1 Biblioteca padrão

O padrão do projeto é usar **Zod** para validações estruturais e de regra.

### 3.2 Organização dos validadores

Por convenção, os schemas ficam em:

- `src/modules/<modulo>/domain/validators`

Exemplo de nome:

- `createProdutoValidator`
- `updateProdutoValidator`
- `conviteValidator`

### 3.3 Como aplicar

Fluxo recomendado:

1. definir schema em `domain/validators`;
2. chamar `parse`/`safeParse` na entidade, value object, policy ou service;
3. lançar erro de domínio (`ValidationDomainError`) quando necessário.

## 4. Exemplo correto

```ts
// domain/validators/validators.ts
export const createProdutoValidator = z.object({
  nome: z.string().min(1, "O nome é obrigatório"),
  valor: z.number().positive("O valor deve ser maior que zero"),
  empresaId: z.string().min(1, "Id da empresa obrigatório"),
});

// domain/entities/Produto.ts
static create(props: CreateProdutoInput): Produto {
  const resolved = {
    ...props,
    valor: Money.create(props.valor),
    dataCriacao: new Date(),
    dataAtualizacao: new Date(),
  };

  const validated = createProdutoValidator.parse(resolved);
  return new Produto(validated);
}
```

Pontos positivos:

- validação centralizada em schema reutilizável;
- entidade protege a própria criação;
- objeto só nasce em estado válido.

## 5. Exemplo incorreto

```ts
// application/use-cases/CreateProdutoUseCase.ts
async execute(input: any) {
  // valida só no use case
  if (!input.nome) throw new Error("Nome obrigatório");

  // domínio recebe dado cru sem validação formal
  const produto = new Produto(input);
  await this.repository.create(produto);
}
```

Problemas:

- regra crítica fica fora do domínio;
- validação ad-hoc e inconsistente;
- risco de entidade nascer em estado inválido;
- duplicação de regra em múltiplos casos de uso.

## 6. Regras práticas

1. toda criação de entidade deve passar por validação de domínio;
2. toda atualização deve preservar invariantes;
3. validações reutilizáveis devem ficar em `domain/validators`;
4. erro de regra de negócio deve ser erro de domínio;
5. validação de transporte (HTTP, DTO) não substitui validação de domínio.

## 7. Checklist rápido

1. a entidade/value object valida no ponto de entrada?
2. as regras estão centralizadas em schemas reutilizáveis?
3. há separação entre validação de entrada (API) e validação de domínio?
4. os erros retornados são coerentes com regra de negócio?
5. existe risco de criar estado inválido por caminho alternativo?
