# Application Validation

## 1. O que é validação de aplicação

Validação de aplicação é a validação de **fluxo e pré-condição de execução**.

Ela acontece na camada de aplicação para garantir que uma operação pode continuar antes de acionar ou persistir mudanças.

## 2. Para que serve

No Jiffy, essa validação serve para:

1. validar pré-condições entre módulos/entidades (ex.: existência de registros);
2. impedir inconsistências de fluxo (ex.: recurso já processado);
3. aplicar checagens de autorização no contexto do caso de uso;
4. proteger integração entre serviços e repositórios.

## 3. Diferença para validação de domínio

### 3.1 Validação de aplicação

- foco em **fluxo de execução**;
- pode depender de consulta a repositório/serviço;
- ocorre em use case ou application service.

### 3.2 Validação de domínio

- foco em **invariantes de negócio**;
- deve viver em entidade/value object/policy/domain service;
- independe de protocolo externo.

## 4. Padrões no Jiffy

### 4.1 Validação de entrada reaproveitada

A aplicação também pode possuir schemas em `application/validators` para inputs específicos de casos de uso.

### 4.2 Validação de pré-condição no fluxo

Casos comuns:

- verificar se entidade relacionada existe;
- evitar duplicidade de operação;
- garantir contexto mínimo para executar a ação.

## 5. Exemplo correto

```ts
export class CreateConviteUseCase {
  async execute(input: CreateConviteInput): Promise<CreateConviteResult> {
    const perfilGestor = await this.pessoaRepository.getPerfilGestorById(
      input.perfilGestorId
    );

    if (!perfilGestor) throw new NotFoundError("Perfil Gestor");

    const conviteExistente = await this.conviteRepository.find({
      email: input.email,
      empresaId: input.empresaId,
      status: "PENDENTE",
      limit: 1,
    });

    if (conviteExistente[0]) {
      throw new ConstraintError("Já existe convite pendente para este e-mail");
    }

    // segue fluxo
  }
}
```

## 6. Exemplo incorreto

```ts
export class CreateConviteUseCase {
  async execute(input: CreateConviteInput) {
    // regra de domínio central no use case
    if (!input.email.includes("@")) throw new Error("Email inválido");

    // criação de entidade ignorando validação de domínio
    return new ConviteEmpresa(input as any);
  }
}
```

Problemas:

- validação sintática duplicada fora do padrão;
- invariantes de domínio não estão protegidas no lugar correto;
- mistura responsabilidades entre aplicação e domínio.

## 7. Checklist rápido

1. validação de aplicação está focada em pré-condição/fluxo?
2. invariantes de domínio ficaram no domínio?
3. existe dependência de repositório apenas para checagens de fluxo?
4. erros de aplicação estão claros (`NotFound`, `Constraint`, `Unauthorized`)?
5. validações duplicadas entre controller/aplicação/domínio foram evitadas?
